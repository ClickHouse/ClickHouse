# Redis Demo Implementation Issues

## Browser Opened Redis Endpoint Port Directly

- Stage: Defense demo / manual verification
- Symptom: Browser showed `-ERR protocol error`.
- Root cause: Port `9006` speaks Redis RESP over TCP, not HTTP.
- Fix: Use Streamlit on port `8501` for the UI; use `redis-cli`, raw RESP, or the demo Redis client for port `9006`.
- Verification: Streamlit UI works on `8501`; `9006` is visible with `ss`; Redis commands work through the demo or `redis-cli`.
- Files affected: Documentation/demo instructions only.

## Streamlit Initially Blank Or Wrong Localhost Target

- Stage: Defense demo startup
- Symptom: Browser page was blank or not usable.
- Root cause: Remote SSH/server access and correct URL/port-forwarding confusion.
- Fix: Open the Streamlit Network URL or use VS Code port forwarding / SSH tunnel.
- Verification: Streamlit app loaded successfully in browser.
- Files affected: Documentation/demo instructions only.

## Earlier Core Issue Cross-Reference

### WriteBuffer Finalization Assertion

- Stage: Redis TCP skeleton.
- Symptom: Server assertion/finalization issue on connection handling.
- Root cause: `WriteBuffer` finalization/cancel path was not handled correctly.
- Fix: `AutoCanceledWriteBuffer` and `finalize` on normal paths.
- Verification: `PING`/`QUIT`/manual Redis checks no longer assert and server stays alive.
- Files affected: Core Redis TCP skeleton implementation.
- Cross-reference: Documented in `project_notes/redis_handler_skeleton_status.md`, where the status note records that the first Redis `PING` response worked but the server terminated due to a `WriteBuffer` finalization assertion, fixed by `AutoCanceledWriteBuffer<WriteBufferFromPocoSocket>` and finalizing normal close paths.
