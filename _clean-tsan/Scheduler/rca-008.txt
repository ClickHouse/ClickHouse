## Alert Analysis

**Error Type:** Livelock (hang with 42.8% CPU)

**Test:** SchedulerWorkloadResourceManager.MemoryReservationIncreaseOfRunningHasPriorityOverPending

**Root Cause:** The test is single-threaded. After killAllocation sets kill_reason and cv.notify_all() (iteration 006 fix), nobody enqueues the decrease because:
- killAllocation can't call performSyncAction (would create activation_mutex → EventQueue::mutex cycle — confirmed real, not false positive: processActivation acquires activation_mutex while STILL holding EventQueue::mutex at line 198, releases mutex at line 199)
- waitKilled (iteration 007 fix) only runs when test thread explicitly calls it, but test thread may be blocked waiting on a different allocation

The production model works because each allocation has its own user thread running syncWithMemoryTracker. The test has one thread.

**Proposed Fix:** In killAllocation, instead of calling performSyncAction directly (which acquires EventQueue::mutex under activation_mutex), use EventQueue::enqueue to post a deferred task that calls performSyncAction. The enqueued task runs from EventQueue::processEvent, which runs OUTSIDE both mutexes (see line 207-208: lock.unlock() before task()). This is safe because:
1. The task runs on the scheduler thread (same thread that calls killAllocation) but after processActivation returns
2. No lock-ordering issues — the task runs with no EventQueue mutexes held
3. The decrease is enqueued promptly, unblocking other allocations

Implementation:
```cpp
void killAllocation(const std::exception_ptr & reason) override
{
    ResourceCost pending_increase = 0;
    ResourceCost pending_decrease = 0;
    {
        std::unique_lock lock(mutex);
        kill_reason = reason;
        real_size = 0;
        computeSyncAction(pending_increase, pending_decrease);
        cv.notify_all();
    }
    // Cannot call performSyncAction here — we're inside EventQueue::activation_mutex scope.
    // Defer to an event that runs outside both EventQueue mutexes.
    if (pending_increase > 0 || pending_decrease > 0)
    {
        queue.event_queue->enqueue({.task = [this, pending_increase, pending_decrease]
        {
            performSyncAction(pending_increase, pending_decrease);
        }});
    }
}
```
