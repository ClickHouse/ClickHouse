## Alert Analysis

**Error Type:** lock-order-inversion (2-mutex cycle within EventQueue)

**Mutexes:** EventQueue::mutex (M0) and EventQueue::activation_mutex (M1)

**Cycle:**
1. Forward: EventQueue::process holds M0, processActivation acquires M1
2. Reverse: processActivation (holding M1) -> AllocationQueue::processActivation -> propagate -> AllocationLimit::setIncrease -> TestAllocation::killAllocation -> performSyncAction -> queue.decreaseAllocation -> scheduleActivation -> enqueueActivation acquires M0

**Root Cause:** TestAllocation::killAllocation calls performSyncAction (which calls queue operations that trigger scheduleActivation), but killAllocation is called from within the scheduler's processActivation chain which holds EventQueue::activation_mutex. Production code (MemoryReservation::killAllocation) avoids this by only setting flags and calling cv.notify_all() — no queue operations.

Note: increaseApproved/decreaseApproved are safe because AllocationQueue sets skip_activation=true before calling them, suppressing scheduleActivation. But killAllocation is called from a different path (AllocationLimit::setIncrease) where skip_activation is NOT set.

**Proposed Fix:** TestAllocation::killAllocation should match MemoryReservation::killAllocation:
- Just set kill_reason, real_size = 0, cv.notify_all()
- Do NOT call computeSyncAction/performSyncAction
- The destructor (running on test thread, outside activation_mutex) handles the decrease
