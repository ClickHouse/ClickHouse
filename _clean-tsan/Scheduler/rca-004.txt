## Alert Analysis

**Error Type:** Hang (lost wakeup / starvation -- classified as livelock with 9.3% CPU)

**Threads/Mutexes Involved:**
- 10 user threads (threads #9-#18, name `exe`): blocked in `condition_variable::wait` inside `DB::MemoryReservation::~MemoryReservation` at `MemoryReservation.cpp:91`, waiting for `allocated_size == 0`
- 1 scheduler thread (thread #7, name `TestScheduler`): blocked in `condition_variable::wait` inside `DB::EventQueue::process` at `EventQueue.cpp`, waiting for events/activations
- 1 main test thread (thread #1): blocked in `pthread_join` at `gtest_space_shared_scheduler.cpp:298`, waiting for user threads to finish
- `AllocationQueue::mutex` (std::recursive_mutex): involved in the `skip_activation` mechanism

**Source Locations:**
- `src/Common/Scheduler/MemoryReservation.cpp:91` -- destructor `cv.wait` for `allocated_size == 0`
- `src/Common/Scheduler/EventQueue.cpp:125` -- scheduler `pending.wait` for events
- `src/Common/Scheduler/Nodes/SpaceShared/AllocationQueue.cpp:242` -- `approveIncrease` second locked section (only `setIncrease`, missing `setDecrease`)
- `src/Common/Scheduler/Nodes/SpaceShared/AllocationQueue.cpp:299` -- `approveDecrease` second locked section (only `setDecrease`, missing `setIncrease`)

**Field/Variable/Mutex:** `AllocationQueue::skip_activation` flag and the `decrease`/`increase` request pointers on `ISpaceSharedNode`

**Current Protection:** The `skip_activation` flag prevents `scheduleActivation` from being called by `increaseAllocation`/`decreaseAllocation` during the callback window in `approveIncrease`/`approveDecrease`. After the callback, only the same-type request pointer is updated (`setIncrease` in `approveIncrease`, `setDecrease` in `approveDecrease`). Cross-type requests are not detected.

**Root Cause:** When the scheduler thread executes `AllocationQueue::approveIncrease`, it sets `skip_activation = true` before calling the `increaseApproved` callback. During this window, if a user thread calls `decreaseAllocation` (e.g., from `~MemoryReservation`), the decrease request is added to `decreasing_allocations` but `scheduleActivation` is suppressed by the `!skip_activation` check. After the callback, the second locked section only calls `setIncrease()` -- it never calls `setDecrease()`, so the `AllocationQueue::decrease` pointer remains `nullptr`. The parent hierarchy (`AllocationLimit`, `SpaceSharedScheduler`) never learns about the decrease request. With no pending activation and `decrease == nullptr` at the scheduler root, the scheduler blocks forever in `EventQueue::process`. All user threads are stuck in their destructors waiting for the scheduler to process their decrease requests.

The same issue applies symmetrically to `approveDecrease` (orphaning increase requests), though the test primarily triggers the `approveIncrease` path.

**Proposed Fix:**

In `AllocationQueue::approveIncrease`, after the callback and re-locking:
```cpp
// After (fixed)
    bool need_activation = false;
    {
        std::lock_guard lock(mutex);
        skip_activation = false;
        increase = nullptr;
        setIncrease();
        if (!decreasing_allocations.empty() && !decrease)
            need_activation = true;
    }
    if (need_activation)
        scheduleActivation();
```

In `AllocationQueue::approveDecrease`, after the callback and re-locking:
```cpp
// After (fixed)
    bool need_activation = false;
    {
        std::lock_guard lock(mutex);
        skip_activation = false;
        decrease = nullptr;
        setDecrease();
        if ((!increasing_allocations.empty() || !pending_allocations.empty()) && !increase)
            need_activation = true;
    }
    if (need_activation)
        scheduleActivation();
```

**Fix Explanation:** The fix detects orphaned cross-type requests by checking if the request lists are non-empty but the corresponding pointer is null (meaning `processActivation` hasn't picked them up yet). If orphaned requests are found, `scheduleActivation` is called outside the lock (maintaining the lock ordering invariant that `AllocationQueue::mutex` must not be held when acquiring `EventQueue::mutex`).
