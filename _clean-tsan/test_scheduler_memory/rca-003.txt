ALERT ID: TSAN-003
TYPE: lock-order-inversion (3-mutex cycle)
SEVERITY: High
STATUS: FIXED

==== ALERT TEXT ====
WARNING: ThreadSanitizer: lock-order-inversion (potential deadlock) (pid=...)

Cycle in lock order graph: M0/0x724c00101328 (MemoryReservation::mutex) => M1/0x7254000d0878 (EventQueue::mutex) => M2/0x7254000d08a0 (EventQueue::activation_mutex) => M0

Edge M0 -> M1 (MemoryReservation -> EventQueue):
  T247 'PipelineExecutor':
    #0  pthread_mutex_lock
    #1  DB::MemoryReservation::syncWithMemoryTracker()
    #2  ... (syncWithScheduler -> increaseAllocation -> scheduleActivation -> enqueueActivation)

Edge M1 -> M2 (EventQueue::mutex -> EventQueue::activation_mutex):
  T249 'SpaceSharedScheduler':
    #0  EventQueue::processActivation

Edge M2 -> M0 (EventQueue::activation_mutex -> MemoryReservation::mutex):
  T249 'SpaceSharedScheduler':
    #0  AllocationQueue::processActivation -> propagate -> AllocationLimit::setIncrease -> MemoryReservation::killAllocation

==== ROOT CAUSE ANALYSIS ====

Three mutexes:
  M_res = MemoryReservation::mutex   (per-allocation state)
  M_eq  = EventQueue::mutex          (event/activation queues)
  M_act = EventQueue::activation_mutex (serializes activation processing)

Lock-order edges:
  M_res -> M_eq:  User thread (PipelineExecutor, TCPHandler)
    syncWithMemoryTracker (holds M_res)
      -> syncWithScheduler
        -> increaseAllocation / decreaseAllocation
          -> scheduleActivation
            -> EventQueue::enqueueActivation (acquires M_eq)

    Same edge from ~MemoryReservation (holds M_res)
      -> decreaseAllocation -> scheduleActivation -> enqueueActivation

  M_eq -> M_act:  Scheduler thread
    EventQueue::processActivation: acquires M_eq then M_act

  M_act -> M_res:  Scheduler thread
    EventQueue::processActivation (holds M_act)
      -> node.processActivation() = AllocationQueue::processActivation
        -> propagate
          -> AllocationLimit::setIncrease
            -> MemoryReservation::killAllocation (acquires M_res)

Cycle: M_res -> M_eq -> M_act -> M_res

Note: scheduler-thread paths through increaseApproved/decreaseApproved are SAFE
because skip_activation = true prevents scheduleActivation() from firing inside
syncWithScheduler during those callbacks. The problem is ONLY in user-thread paths.

==== FIX APPLIED ====

Files modified: src/Common/Scheduler/MemoryReservation.cpp

Strategy: Two-phase lock pattern for user-thread paths in ~MemoryReservation and
syncWithMemoryTracker. Break the M_res -> M_eq edge by releasing M_res before
calling queue operations.

Phase 1 (under M_res): Determine action, set flags (increase_enqueued / decrease_enqueued),
  capture sizes.
Phase 2 (M_res released): Call queue.increaseAllocation() / queue.decreaseAllocation()
  which may call scheduleActivation() -> enqueueActivation() acquiring M_eq.
Phase 3 (re-acquire M_res): cv.wait() for completion.

After fix, lock-order edges involving M_res from user threads are eliminated.
The remaining graph (M_eq -> M_act -> M_res for scheduler thread only) is acyclic.

syncWithScheduler() is now only called from increaseApproved/decreaseApproved
(scheduler thread, skip_activation=true active), which is safe.

Safety of two-phase approach:
- increase_enqueued / decrease_enqueued flags are set before releasing M_res,
  preventing duplicate enqueues from concurrent syncWithScheduler calls.
- cv.wait re-checks predicate on re-acquire, handling the case where
  increaseApproved/decreaseApproved fires between phase 2 and phase 3.
- In ~MemoryReservation: no pending increase_enqueued at destruction time
  (constructor waits for initial approval, syncWithMemoryTracker waits for approval
  before returning -- so the user thread cannot be in both simultaneously).
