## Description {#description}

This table contains profiling on processors level (that you can find in [`EXPLAIN PIPELINE`](../../sql-reference/statements/explain.md#explain-pipeline)).

## Columns {#columns}

<!--AUTOGENERATED_START-->
- `hostname` ([LowCardinality(String)](/sql-reference/data-types/lowcardinality)) вҖ” Hostname of the server executing the query.
- `event_date` ([Date](/sql-reference/data-types/date)) вҖ” The date when the event happened.
- `event_time` ([DateTime](/sql-reference/data-types/datetime)) вҖ” The date and time when the event happened.
- `event_time_microseconds` ([DateTime64(6)](/sql-reference/data-types/datetime64)) вҖ” The date and time with microseconds precision when the event happened.
- `id` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” ID of processor.
- `parent_ids` ([Array(UInt64)](/sql-reference/data-types/array)) вҖ” Parent processors IDs.
- `plan_step` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” ID of the query plan step which created this processor. The value is zero if the processor was not added from any step.
- `plan_step_name` ([String](/sql-reference/data-types/string)) вҖ” Name of the query plan step which created this processor. The value is empty if the processor was not added from any step.
- `plan_step_description` ([String](/sql-reference/data-types/string)) вҖ” Description of the query plan step which created this processor. The value is empty if the processor was not added from any step.
- `plan_group` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” Group of the processor if it was created by query plan step. A group is a logical partitioning of processors added from the same query plan step. Group is used only for beautifying the result of EXPLAIN PIPELINE result.
- `initial_query_id` ([String](/sql-reference/data-types/string)) вҖ” ID of the initial query (for distributed query execution).
- `query_id` ([String](/sql-reference/data-types/string)) вҖ” ID of the query.
- `name` ([LowCardinality(String)](/sql-reference/data-types/lowcardinality)) вҖ” Name of the processor.
- `elapsed_us` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” Number of microseconds this processor was executed.
- `input_wait_elapsed_us` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” Number of microseconds this processor was waiting for data (from other processor).
- `output_wait_elapsed_us` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” Number of microseconds this processor was waiting because output port was full.
- `input_rows` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” The number of rows consumed by processor.
- `input_bytes` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” The number of bytes consumed by processor.
- `output_rows` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” The number of rows generated by processor.
- `output_bytes` ([UInt64](/sql-reference/data-types/int-uint)) вҖ” The number of bytes generated by processor.
- `processor_uniq_id` ([String](/sql-reference/data-types/string)) вҖ” The uniq processor id in pipeline.
- `step_uniq_id` ([String](/sql-reference/data-types/string)) вҖ” The uniq step id in plan.
- `memory_usage_delta` ([Int64](/sql-reference/data-types/int-uint)) вҖ” Signed difference between memory allocated and freed while executing processor work. It excludes processor prepare, scheduling and destruction, and can be negative.
<!--AUTOGENERATED_END-->

Here you can see:

- `ExpressionTransform` was executing `sleep(1)` function, so it `work` will takes 1e6, and so `elapsed_us` > 1e6.
- `SourceFromSingleChunk` need to wait, because `ExpressionTransform` does not accept any data during execution of `sleep(1)`, so it will be in `PortFull` state for 1e6 us, and so `output_wait_elapsed_us` > 1e6.
- `LimitsCheckingTransform`/`NullSource`/`LazyOutputFormat` need to wait until `ExpressionTransform` will execute `sleep(1)` to process the result, so `input_wait_elapsed_us` > 1e6.
